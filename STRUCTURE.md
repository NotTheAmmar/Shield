# SHIELD Architecture & Repository Structure

Welcome to the SHIELD (Secure Hash-based Immutable Evidence Locker & Database) project. This document explains the architecture of our system and the layout of this monorepo.

## 1. System Architecture

The SHIELD system is designed to provide secure, immutable, and verifiable storage for First Information Reports (FIRs) and digital evidence. To achieve this, we use a decentralized, microservices-based approach.

### Core Modules

1.  **Frontend (`shield-frontend`)**: The user-facing web application built with Next.js. This is where users submit FIR details and upload digital evidence.
2.  **API Gateway (`shield-gateway`)**: A Next.js based BFF (Backend-for-Frontend) that routes requests from the frontend to the appropriate microservices.
3.  **Authentication Service (`shield-auth`)**: A Node.js service responsible for user administration, authentication, and authorization. It utilizes PostgreSQL for user data storage.
4.  **Evidence Service (`shield-evidence`)**: A Node.js service that handles the core business logic of hashing the evidence (SHA-256) upon upload and storing the raw files securely in MinIO (an S3-compatible object store).
5.  **Ledger Service (`shield-ledger`)**: A Node.js service that interfaces with Immudb. It stores the cryptographic hashes, timestamps, and case identifiers generated by the Evidence Service, creating an immutable, cryptographically verifiable chain of custody.

### Infrastructure Layer

The project relies on three core databases/stores, managed via Docker:
-   **PostgreSQL (`db-users`)**: Stores relational user data and session information for the Auth service.
-   **Immudb (`db-ledger`)**: A lightweight, high-speed immutable database used to store the cryptographic hashes (SHA-256) of all evidence and FIR records. This provides the tamper-evident guarantees of the system.
-   **MinIO (`minio-store`)**: Secure, off-chain storage for the actual raw digital evidence files (videos, documents, images).

## 2. Directory Structure

This project uses a **Monorepo** structure. All microservices and infrastructure configurations live in this single repository to simplify collaboration and testing.

```text
shield-project/
├── .gitignore             # Global gitignore files (Recursive)
├── .dockerignore          # Global dockerignore
├── README.md              # Main project documentation and run guide
├── CONTRIBUTING.md        # Guidelines for collaborating on this repo
├── package.json           # Root scripts for cross-platform task running
├── docker-compose.yml     # Main composition file orchestrating all services and databases
├── .docker-data/          # [GitIgnored] Local persistence volumes for PostgreSQL, Immudb, and MinIO
├── .github/               # GitHub issue and PR templates
├── .vscode/               # VS Code Workspace settings (Debug configs & Tasks)
│
├── shield-frontend/       # [Next.js] User Interface
├── shield-gateway/        # [Next.js] API Gateway / BFF
├── shield-auth/           # [Node.js] Authentication Service
├── shield-evidence/       # [Node.js] Evidence Management
└── shield-ledger/         # [Node.js] Ledger Interface
```

## 3. Local Development Environment

We use Docker Compose to orchestrate the entire environment locally. Through our root `package.json`, we provide cross-platform scripts to easily spin up all 5 node services alongside the 3 infrastructure containers on a custom Docker bridge network (`shield-network`).

Please refer to the `README.md` for exact instructions on how to install dependencies and boot the application cluster.
